Case study

Building a Graph-Based Warehouse Optimization and Simulation Workflow

I built a warehouse graph construction tool and a simplified simulation model to evaluate how inventory placement changes could reduce picker travel distance, and I am continuing to extend the system toward more realistic operational analysis.

Starting Point

The warehouse had node data, but not a usable graph.

We had storage node information such as coordinates, but no practical graph that represented how workers could actually move through the warehouse. That made it difficult to reason about layout, access, and inventory placement in a structured way.

The immediate problem was not optimization yet. It was building the path-aware model needed before any meaningful simulation or routing analysis could happen.

Current Scope

This is a graph construction tool plus a simplified simulation.

I am not presenting this as a finished warehouse optimization engine. The current system has two parts: a tool for constructing the warehouse graph, and a simplified simulation model for estimating the impact of inventory reallocation.

The goal is directional decision support, not a perfect digital twin.

Graph Construction

Turn storage nodes and walkable paths into an analyzable warehouse graph

Node grouping

Storage nodes are grouped first. The current approach uses Manhattan distance to form practical groups before routing is computed at a larger scale.

Access points

Each group is assigned one or more access points so the graph can represent how workers enter and interact with that part of the warehouse.

Path-based connections

Users draw walkable paths, and the tool automatically connects nearby access points to those paths to generate a usable graph structure.

Flow

How the warehouse model is constructed

flowchart TD A[Storage Nodes] --> B[Node Grouping] B --> C[Access Points] C --> D[Walkable Paths] D --> E[Warehouse Graph] E --> F[Shortest Path Logic] F --> G[Simulation]

The graph construction step comes first. Simulation and routing only become useful after the warehouse can be represented as a usable movement model.

Why Grouping Matters

Reduce routing cost before trying to optimize the warehouse

Running shortest-path calculations directly across every storage node would be expensive. Grouping reduces graph complexity so the routing problem becomes more tractable. That makes later simulation and optimization work more realistic from a computation and workflow perspective.

In practice, the grouping layer is not just a data cleanup step. It is part of the system design for controlling graph size and path computation cost.

Simulation

A simplified model for testing inventory reallocation ideas

Rule-based ranking

Products and storage nodes are ranked using a simplified grading logic instead of a full operational model.

Reallocation test

Inventory is reassigned according to those grades, and the model compares estimated picker travel distance before and after reallocation.

Directional output

The result is meant to explain whether better placement should materially reduce movement, not to claim an exact final operating result.

~50%

Estimated travel distance reduction in the simplified simulation

Not final

The result is directional and does not fully reflect real warehouse constraints

Key risk

Local bottlenecks may appear if picking activity concentrates around specific locations

Interpretation

The value is in the explanation, not just the number.

Promising signal

The simplified simulation suggests that inventory placement can have a large effect on travel distance.

Important limitation

I do not treat the 50 percent figure as a final operational claim because the current model does not fully capture congestion or other real execution constraints.

Main takeaway

Even at this stage, the graph-based model helps explain in a structured way why better inventory placement should materially reduce movement.
