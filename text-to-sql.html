Case study

Building an AI-Powered Text-to-SQL System to Scale Data Access

I turned repeated data research into a reusable knowledge system for AI, reducing ad-hoc analysis time from 4 to 10 hours to under 1 hour.

Context

The bottleneck was understanding, not SQL syntax.

Ad-hoc requests were consuming too much analyst time. The repeated work was not just query writing. It was identifying the right tables, interpreting column meaning, mapping business logic correctly, and figuring out join patterns that could be trusted.

That made even routine questions expensive. Complex requests often took 4 to 10 hours, which limited how many requests the team could realistically support each month.

Approach

Make the knowledge reusable before trying to automate the SQL.

I built a structured knowledge layer in YAML that captured how analysts actually reason about the data model: table definitions, column semantics, keys, partitioning, join patterns, and business terminology.

The leverage came from structuring context well enough that AI agents could use it repeatedly, not from treating the model as magic.

What the system did

A practical validation loop around AI-generated SQL

Scoped context

Tables were grouped by domain and narrowed through index files so the agent only saw the relevant slice of metadata.

SQL generation

AI coding agents used the structured metadata to generate and revise SQL based on the actual business question.

Execution and QA

The workflow ran queries in Athena, checked execution success, and validated results before handing anything back to a human.

System Flow

From question to validated answer

flowchart TD A[User Question] --> B[Domain and Table Selection] B --> C[Knowledge Layer] C --> C1[Table Definitions] C --> C2[Column Semantics] C --> C3[Join Patterns] C --> C4[Business Terms] C --> D[AI Agent] D --> E[SQL Query] E --> F[Run in Athena] F --> G{Execution OK} G -- No --> D G -- Yes --> H[Result Validation] H --> I{Quality OK} I -- No --> D I -- Yes --> J[Deliver to Human] J --> K[Update Knowledge] K --> C

The system did not stop at generation. It executed, checked, and iterated until the result was reliable enough for review.

What I learned

The real system was the knowledge architecture.

Understanding is the hard part

Most of the cost in analytics work comes from reconstructing how the business maps onto the data model every single time.

Simple structure scales

A clean YAML knowledge layer plus execution and QA created more leverage than a heavier platform approach would have in this setting.

AI works best with discipline

The output becomes useful when the workflow is constrained, validated, and tied back into reusable team knowledge.
