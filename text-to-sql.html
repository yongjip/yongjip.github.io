Building an AI-Powered Text-to-SQL System to Scale Data Access

I turned repeated data research into a reusable knowledge system for AI, reducing ad-hoc analysis time from hours to under one hour.

Context

In our data team, ad-hoc requests were a recurring bottleneck. Simple requests could be answered quickly, but more complex questions often required 4–10 hours of research and query writing.

The hardest part was usually not writing SQL itself. Most of the time went into understanding which tables to use, how to join them correctly, what each column meant, and how business logic should be reflected in the query.

Problem

To answer a single question, analysts often had to:

  • identify relevant tables across the data lake
  • understand column semantics and data relationships
  • figure out reliable join patterns
  • apply business-specific definitions correctly

This process was repetitive, time-consuming, and hard to scale.

Idea

Instead of trying to make humans write SQL faster, I focused on making the knowledge behind query writing reusable and accessible to AI.

The key is not just the model. It is how the problem is structured.

Solution

I built a Text-to-SQL system powered by AI coding agents such as Codex, Claude Code, and Gemini.

The core of the system is a structured knowledge layer that captures how analysts actually explore data:

  • table definitions
  • column meanings
  • partition and key information
  • common join patterns
  • business terminology mappings

This metadata was stored in YAML so AI agents could use it as context when generating queries.

To reduce context size and improve efficiency, I grouped tables by domain and used index files to narrow down the relevant metadata before passing it to the model.

System Flow

flowchart TD A[User Question] --> B[Domain / Table Selection] B --> C[Structured Knowledge Layer] C --> C1[Table Definitions] C --> C2[Column Semantics] C --> C3[Join Patterns] C --> C4[Business Terms] C --> D[AI Agent (Codex / Claude / Gemini)] D --> E[SQL Query Generation] E --> F[Run Query in Athena] F --> G{Syntax / Execution OK?} G -- No --> D G -- Yes --> H[Result QA / Data Validation] H --> I{Quality Check Passed?} I -- No --> D I -- Yes --> J[Deliver to Human] J --> K[Use Result / Update Knowledge if Needed] K --> C

The system does not stop at SQL generation. It executes, checks, and iterates until the output is reliable enough for human review.

Workflow

  1. A user asks a question.
  2. The system identifies relevant domains and tables.
  3. Structured metadata is retrieved.
  4. An AI agent generates SQL.
  5. The query is executed in Athena.
  6. If there is a syntax or execution issue, the query is revised.
  7. Results go through QA and data validation.
  8. Once the output looks reliable, it is delivered to a human.
  9. New learnings are added back into the knowledge layer.

Impact

  • Reduced ad-hoc query time from 4–10 hours to under 1 hour
  • Increased monthly query capacity from around 20 to 50+
  • Shifted human effort from manual query writing to validation and decision-making

What I Learned

  • Most of the difficulty in data work comes from understanding, not computation.
  • Structuring knowledge is more scalable than repeating research manually.
  • Simple systems, when designed well, can outperform more complex solutions.
