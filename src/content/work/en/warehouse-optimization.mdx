<DetailSection
  label="Overview"
  title="Warehouse optimization started with building a graph the team could actually analyze."
  class="detail-section-results"
>
  <DetailColumns>
    <DetailItem title="Starting point">
      <p>We had storage node information such as coordinates, but no practical graph that represented how workers could actually move through the warehouse. That made it difficult to reason about layout, access, inventory placement, or picker travel distance in a structured way.</p>
      <p>The immediate problem was not optimization yet. It was building the path-aware model needed before any meaningful simulation or routing analysis could happen.</p>
    </DetailItem>
    <DetailItem title="Current scope">
      <p>I am not presenting this as a finished warehouse optimization engine. The current system has two parts: a tool for constructing the warehouse graph, and a simplified simulation model for estimating how inventory reallocation changes picker travel distance.</p>
      <p>The goal is directional decision support, not a perfect digital twin.</p>
    </DetailItem>
  </DetailColumns>
</DetailSection>

<DetailSection
  label="Graph construction"
  title="Turn storage nodes and walkable paths into an analyzable warehouse graph."
  class="detail-section-takeaways"
  intro="The graph comes first. Once movement is represented cleanly, routing and simulation become possible on top of it."
>
  <DetailStack>
    <DetailItem title="Node grouping">
      <p>Storage nodes are grouped first. The current approach uses Manhattan distance to form practical groups before routing is computed at a larger scale.</p>
    </DetailItem>
    <DetailItem title="Access points">
      <p>Each group is assigned one or more access points so the graph can represent how workers enter and interact with that part of the warehouse.</p>
    </DetailItem>
    <DetailItem title="Path-based connections">
      <p>Users draw walkable paths, and the tool automatically connects nearby access points to those paths to generate a usable graph structure.</p>
    </DetailItem>
    <DetailItem title="Why grouping matters">
      <p>Running shortest-path calculations directly across every storage node would be expensive. Grouping reduces graph complexity so later routing and simulation work remains tractable.</p>
    </DetailItem>
  </DetailStack>
</DetailSection>

<DetailSection label="Simulation" title="A simplified model for testing inventory reallocation ideas.">
  <DetailStack>
    <DetailItem title="Rule-based ranking">
      <p>Products and storage nodes are ranked using a simplified grading logic instead of a full operational model.</p>
    </DetailItem>
    <DetailItem title="Reallocation test">
      <p>Inventory is reassigned according to those grades, and the model compares estimated picker travel distance before and after reallocation.</p>
    </DetailItem>
    <DetailItem title="Directional output">
      <p>The result is meant to explain whether better placement should materially reduce movement, not to claim an exact final operating result.</p>
    </DetailItem>
  </DetailStack>
</DetailSection>
